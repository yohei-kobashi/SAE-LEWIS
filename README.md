# SAE-LEWIS

A sparse-autoencoder–conditioned variant of LEWIS for controllable, feature-level text editing. The intervention spec over SAE features takes the role of LEWIS's discrete style label and conditions every stage of the pipeline through a single shared projection. Training is self-supervised on a single monolingual corpus.

## 1. Motivation

### 1.1 Refining SAE-based intervention

Sparse Autoencoders (SAEs) decompose LLM hidden states into a high-dimensional, sparsely activated basis where each direction tends to capture a single interpretable concept. This has motivated a line of work on **direct causal intervention**: at inference, the SAE encodes a residual stream activation, a target feature's value is overwritten (ablated to 0 or enhanced to a fixed positive constant), the SAE decoder reconstructs the residual, and forward propagation continues. **LinguaLens** (Jing et al., 2025) operationalizes this paradigm at scale for linguistic mechanism interpretation: it catalogs roughly 145 SAE features across morphology, syntax, semantics, and pragmatics, identifies the base vectors causally associated with each feature via Probability-of-Sufficiency / Necessity / FRC, and demonstrates statistically significant intervention effects on Llama-3.1-8B.

LinguaLens also explicitly states the limitation of the direct-activation approach: *"although our experiments show statistically significant effects from feature-based interventions, the efficacy and stability of single interventions remain inferior to conventional fine-tuning techniques. This shortcoming calls for further research to refine SAE-based intervention methods."* The same paper observes empirically that ablating a single feature is often compensated by parallel feature pathways at the same and neighbouring layers, so the intervention's effect on the model's output is weaker than its causal signature inside the network.

Three properties of the direct-activation paradigm contribute to this gap:

1. **Pathway dilution.** A single linguistic phenomenon is represented by multiple correlated features across multiple layers; modifying one feature in one layer can be compensated by parallel pathways.
2. **Unprincipled magnitude.** The choice of an ablation target (0) or an enhancement target (10) is a hyperparameter; the operationally correct magnitude depends on context, feature, and layer.
3. **Off-manifold residuals.** Aggressive activation overwrites shift the residual stream away from the distribution the LLM was trained on, degrading downstream fluency.

SAE-LEWIS is proposed as a refinement of this paradigm.

### 1.2 SAE specification as editor conditioning

Instead of modifying SAE activations inside the LLM's forward pass, SAE-LEWIS routes the SAE specification through a **separate, controllable text editor**. The user-supplied intervention — a non-negative sparse vector over SAE features — conditions a tagger and an editor that together produce a discrete sequence of edit operations (KEEP / REPLACE / INSERT / DELETE) on the input text. The intervention is realized as a token-level diff. The original LLM, SAE, and LM head are not modified at inference: there is no in-pass activation surgery.

This indirection addresses the three properties above:

1. The intervention does not have to overpower parallel pathways at any single layer; it is consumed by a dedicated editor that maps SAE specifications to text-level edits.
2. The intervention magnitude is set by the editor's training, not by an unprincipled scalar at inference.
3. The LLM sees in-distribution residuals throughout; output fluency is therefore preserved.

### 1.3 Backbone architecture: LEWIS

We adopt the architecture of LEWIS (Reid & Zhong, 2021), a coarse-to-fine, op-based text editor:

1. A **coarse tagger** that predicts per-token edit operations (KEEP / DELETE / REPLACE / INSERT).
2. A **fine MLM** (BART) that fills in the masked spans, with candidate selection by an external style classifier.

LEWIS conditions both stages on a discrete style label. SAE-LEWIS replaces this label with a non-negative sparse intervention vector over SAE features, projected through a learned linear map `Proj_A`. Training data is generated by MLM-driven corruption of clean text from a monolingual corpus; the conditioning vectors used during training are derived from the SAE forward of the clean and corrupted texts.

```
LEWIS                            SAE-LEWIS (v2)
─────────────────────────        ─────────────────────────────────────────────
discrete style label             non-negative sparse SAE intervention vector
coarse two-tag tagger            SAE-conditioned two-tag tagger
  (binary insert + 3-class op)     (binary insert head + 3-class KEEP/REPL/DEL head)
fine generator over x SEP x_c    SAE-conditioned bidirectional editor over
  (BART, AR)                       x' [SEP] x'_c (LLM2Vec + frozen LM head, non-AR)
deletion decided by tagger,      deletion decided by tagger, DEL tokens
  removed from generator input     removed from the editor template
multi-cand + classifier          template enumeration + SAE-aware ranker
pseudo-parallel pairs            monolingual corpus + MLM-based corruption
                                 (SAE-derived conditioning)
```

The same projection `Proj_A : ℝ^{d_sae} → ℝ^{d_model}` conditions both the tagger and the editor. The underlying LLM, SAE, and LM head are frozen throughout.

**v2 — LEWIS-faithful revision.** The first implementation (v1) deviated from LEWIS in three places, and held-out evaluation traced a concrete failure to each deviation:

1. *4-class tagger with INS on the gap-adjacent token* — the INS tag sat on a token whose content is unchanged (i.e. exactly a KEEP token), so INS competed with KEEP inside one softmax and held-out INS F1 was 0.000 even at support 297. LEWIS instead predicts **two tags per token** (a binary "insert before this token" indicator plus a 3-class non-insertion op, §2.1 of the paper); v2 adopts this two-head design.
2. *Editor input hid the pre-edit word* — v1's editor saw only the template ([MASK] at REPL positions), turning REPL into unconstrained recovery; the conditioning-causality probe scored the editor IGNORED for REPL. LEWIS's fine generator sees **both** the source and the template (`x SEP x_c`, eq. 2); v2 feeds the editor `x' [SEP] x'_c`.
3. *Editor re-decided deletion* — v1 kept DEL tokens in the editor input and supervised a [DEL] output there, duplicating the tagger's decision through a tied logit-column mechanism that the probe flagged as an opaque presence-flag (random conditioning worked almost as well as true). In LEWIS deletion is the tagger's decision alone and the generator never sees deleted tokens; v2 removes DEL tokens from the template and drops the [DEL]-output pathway entirely.

v2 retains the three SAE-LEWIS identity properties unchanged: the LLM2Vec bidirectional encoder, the `z_amp`/`z_sup` two-vector intervention interface, and its injection into the LLM's representation space via `Proj_A` soft-prefix tokens.

### 1.4 Conditioning grounded in SAE features

The training-time conditioning vectors `z_amp` and `z_sup` are computed from the SAE pool-max difference between the clean and corrupted text of each training sample. Every feature placed in the conditioning vector is therefore one that the SAE itself identifies as differing between the two texts. This design has two direct consequences:

1. The semantic role of each conditioning feature is fixed by the SAE — not by which axis of variation happens to dominate a corpus. The model learns to use feature `i` for whatever the SAE encodes feature `i` to mean.
2. The conditioning at inference (a user-specified sparse vector over SAE features) and at training (an SAE-derived sparse vector from the corruption diff) share the same semantics. The interface is consistent across the train / inference boundary.

### 1.5 Fluency-preserving corruption

Both inputs and targets in training are required to be well-formed text. The pipeline applies multiple ops (REPL / INS / DEL) per sentence as a **compound corruption** and gates the entire compound by rejection sampling:

- **MLM plausibility (DEL only)** — for DEL ops a token is inserted only if the MLM predicts it to fit at the chosen position. INS deletes natural words at positions that are easy to remove (modifiers, adverbial phrases, sentence-initial words, function words); no MLM recoverability gate. REPL substitutes via MLM top-K and trains the editor to recover the original word.
- **Perplexity** — the compound's perplexity ratio against the clean text is bounded; the threshold scales with `sqrt(N)` where N is the op count, reflecting the per-op contribution to log-PPL.
- **SAE shift** — bounded both below (the compound must carry usable signal) and above (the compound must remain a minimal pair); both bounds scale with `sqrt(N)`.

Thresholds are calibrated against the empirical Dolma + MLM compound distribution by percentile, so yield is controllable and the pipeline is self-consistent (§6.2.6). The editor is therefore trained on coordinated multi-op edits with controlled aggressiveness, matching the inference-time setting where real edits typically combine several ops.

## 2. End-to-End Pipeline

```

   │
   ├── build (z_amp, z_sup) ── Proj_A + type_emb ──► (INT_amp, INT_sup)
   │                                                        │
   ▼                                                        ▼
[INT_amp, INT_sup, e(text)]                  [INT_amp, INT_sup, e(x'), e([SEP]), e(x'_c)]
   │ LLM2Vec'd Gemma (frozen)                              ▲
   │ two heads (LEWIS two-tag):                            │
   │   op3 ∈ {KEEP, REPL, DEL}   per token                 │
```

## 3. Method Identity (key design choices)

| # | Property |
| --- | --- |
| C1 | Frozen LLM (Gemma-2 2B), frozen SAE (Gemma Scope), frozen LM head, frozen embedding table. The LLM2Vec encoder backbone is adapted with **LoRA** on the attention/MLP projections — LEWIS fully fine-tunes both of its networks, and a fully frozen backbone (the earlier deviation) left only the linear `Proj_A` to route the conditioning prefix to the edit positions. Tagger and editor each get their OWN adapter (LEWIS's tagger and generator are separate networks). LoRA is also the adaptation style the LLM2Vec checkpoint itself was built with (MNTP and SimCSE are LoRA stages). `--lora-r 0` recovers the frozen-backbone ablation. |
| C2 | Tagger and editor share the same conditioning interface (`INT_amp`, `INT_sup`): identical `z` construction and semantics. The editor is trained first; the tagger's `Proj_A` / `type_emb` / `cond_scale` are warm-started from the editor checkpoint and then fine-tuned. |
| C3 | Intervention is encoded as **two non-negative sparse vectors** (`z_amp`, `z_sup`); sign is carried by `type_emb`. |
| C4 | Tagger is **two-tag per token** (LEWIS §2.1): a binary "insert phrase before this token" head plus a 3-class non-insertion op head over KEEP / REPL / DEL. Insertion is a boundary property and never competes with KEEP in a softmax. An insert indicator at position `i` opens a `[INS]` gap to the left of token `i`; the slot count per gap is set by template enumeration (C8). |
| C5 | Editor is a **single bidirectional pass** over the LLM2Vec encoder with the frozen LM head. No autoregressive decoder. |
| C6 | Editor input follows LEWIS eq. 2: the concatenation `x' [SEP] x'_c` of the source text and the edit template (REPL → `[MASK]`, insertion gaps → `[INS]` slots, KEEP → original token, DEL → token removed). The editor sees the pre-edit words it must replace or restore; loss and decoding apply to the `x'_c` segment only. |
| C7 | Edit output length is variable; INS adds slots, DEL shortens the template before the editor runs. Deletion is the tagger's decision alone — the editor emits no special tokens (the v1 `[DEL]`-output pathway is removed). |
| C8 | Multi-token insertion is realised by **template enumeration over slot counts** plus an SAE-aware **ranker**, mirroring LEWIS's "candidates + selector" paradigm. |
| C9 | Training is self-supervised on a monolingual corpus; supervision is generated by fluency-preserving MLM-based corruption of clean sentences. |
| C10 | Training-time `z_amp` and `z_sup` are derived from the SAE pool-max difference between clean and corrupted text, propagating SAE semantics into the conditioning interface. |
| C11 | Conditioning sub-sampling and small per-sample `k_amp` / `k_sup` (uniform over {1..4}) prevent the model from using the diff-based conditioning as a direct shortcut to the target. Zero-conditioning behaviour is supervised by the §6.2.8 `null` records (z = 0 ⇔ target = input); the earlier empty-cond-prob hack — z = 0 with the full-restore target kept — is retired, as it actively taught editing without instruction. |
| C12 | Training data is generated by **compound corruption**: each accepted sample combines N ∈ {0..N_MAX} ops (REPL / INS / DEL) on a single clean sentence, gated by PPL and SAE-shift filters whose thresholds scale with `sqrt(N)`. Single-op (N=1) and identity (N=0) samples are retained as a small fraction for tagger op-discrimination and KEEP supervision. INS deletion positions are biased toward syntactic categories where removal is least disruptive (modifiers, adverbial phrases, sentence-initial words, function words). |
| C13 | Training operates at sentence granularity; corpus documents are sentence-segmented before any further processing. |
| C14 | Trainable parameters: `Proj_A`, the tagger's two small heads (3-class op + binary insert), `type_emb[0..2]`, a learnable conditioning gain `cond_scale`, input-embedding delta rows for `[MASK]` / `[INS]` / `[SEP]`, per-model LoRA adapters on the backbone (C1; rank 16 default, fp32, saved inside the trainable checkpoint), and (ablation) the length predictor head. Embedding table and LM head are never trained (except through the tied delta rows), so editor outputs stay grounded in Gemma's vocabulary geometry. |

## 4. Components

### 4.1 Intervention Encoder (shared)

Builds two non-negative sparse vectors and projects each through the shared `Proj_A`. The vector source differs between inference and training.

**Inference path** (user-provided spec):

```python
z_amp = zeros(d_sae); z_sup = zeros(d_sae)
for f, sign in spec:
    if sign == "+": z_amp[f] = mu[f] * strength
    else:           z_sup[f] = mu[f] * strength

INT_amp = Proj_A(z_amp) + type_emb[1]    # (d_model,)
INT_sup = Proj_A(z_sup) + type_emb[2]    # (d_model,)
```

**Training path** (diff-based, see §6.3.3):

```python
# SAE pool-max top-K for clean and corrupted text
z_X        = sae_pool_max_topk(encode(X),        K=K_train)
z_X_prime  = sae_pool_max_topk(encode(X_prime),  K=K_train)
delta = z_X - z_X_prime

# Candidate features per sign, then sub-sample (information-leak control)
amp_cands = topk_indices(max(delta, 0),  K_top)
sup_cands = topk_indices(max(-delta, 0), K_top)
k_amp = random.choice([1, 2, 3, 4]); k_sup = random.choice([1, 2, 3, 4])
z_amp = sparse(sample(amp_cands, k_amp), values= delta[...])
z_sup = sparse(sample(sup_cands, k_sup), values=-delta[...])
if random() < empty_cond_prob:    # default 0.05
    z_amp = z_sup = sparse_zeros()

INT_amp = Proj_A(z_amp) + type_emb[1]
INT_sup = Proj_A(z_sup) + type_emb[2]
```

- `mu[f]` is the per-feature mean activation, precomputed in `mu.npy` by `precompute_sae.py`.
- Non-negative inputs keep `Proj_A`'s input distribution aligned with the frozen SAE's output distribution.
- **`Proj_A` is grounded in the SAE decoder (v3.1, default `wdec-frozen`).** v3 probes showed the conditioning being USED but random feature indices working almost as well as the true ones (OPAQUE-FLAG): with 16,384 features and only ~10 conditioning occurrences per feature in a 100k-record cache, a random-init linear `Proj_A` cannot learn per-feature identity. But identity does not need to be learned — `W_dec` row `f` IS feature `f`'s direction in the LLM's residual stream, a representation the (same-model) Gemma backbone natively interprets. Modes: `wdec-frozen` (default) fixes `Proj_A = W_dec` and trains only a rank-32 correction `B·A` on top (`B = 0` init → exact `W_dec` at step 0; identity can never be washed out); `wdec-init` uses `W_dec` as a trainable init; `learned` is the pre-v3.1 random-init ablation. This also strengthens identity property #3: the intervention enters the model literally as the amplified/suppressed features' residual-stream directions.
- `type_emb[1]` / `type_emb[2]` mark amp / sup conditioning tokens. `type_emb[0]` is reserved for text tokens but intentionally NOT added to them — adding a constant learned offset to every text embedding of a frozen encoder only risks pushing it off distribution; the two prefix tokens are already distinguishable by their own type embeddings.
- **Scale calibration.** `Proj_A`'s raw output magnitude tracks `‖z‖`, which varies per sample by orders of magnitude (SAE activation deltas are feature-dependent), and Gemma multiplies all `inputs_embeds` by `sqrt(d_model) ≈ 48` internally. Each conditioning vector is therefore RMS-normalized to the median token-embedding row RMS of the checkpoint, times a learnable scalar gain `cond_scale` (init 1.0), before `type_emb` is added. This keeps the prefix from either vanishing against `type_emb` or blowing the frozen encoder off distribution.
- **Prefix layout is `[INT_amp, INT_sup, <bos>, text…]`** — the two soft tokens are prepended in embedding space to token ids that already begin with `<bos>`. With RoPE's relative positions this preserves the exact `<bos>`↔text geometry the LLM2Vec encoder saw during MNTP/SimCSE; placing `<bos>` first instead would shift every text token +2 relative to `<bos>`, which is more out-of-distribution for the frozen encoder.

### 4.2 Tagger

- **Input:** `[INT_amp, INT_sup, e(text_1), …, e(text_T)]`
- **Backbone:** LLM2Vec'd Gemma + own LoRA adapter (attention/MLP projections, rank 16 default; `--lora-r 0` = frozen ablation)
- **Heads (LEWIS two-tag, §2.1 of the paper):**
  - `op_head` — per-token 3-way MLP over `{KEEP, REPL, DEL}` (the non-insertion op for the token itself)
  - `ins_head` — per-token binary MLP: "insert a phrase before this token?" Insertion is a property of the token *boundary* — the token itself is unchanged — so it must not compete with KEEP inside a softmax (the v1 4-class design did exactly that and never predicted INS).
- **Training data:** the corruption stream of §6.2 / §6.3, identical to the editor's data. The cache stores the v1 4-class gold; `lewis_ops.split_cache_tags` converts it losslessly (a cached INS tag always sits on an otherwise-KEEP token).
- **Loss:** class-weighted cross-entropy (op head, compensates KEEP majority) + BCE with `pos_weight` = capped neg/pos ratio (insert head), summed with `--ins-loss-weight` (default 1.0). Per-class counts and the pos_weight are logged at warmup.
- **Warm-start:** `Proj_A` / `type_emb` / `cond_scale` are initialized from the trained editor checkpoint (`--init-proj-a-from`), since the editor's CE through the LM head gives the conditioning projection a much richer gradient than the small-head losses.
- **Warmup:** `Proj_A` is frozen for the first steps (only the heads update), then unfrozen

### 4.3 Editor

- **Input:** `[INT_amp, INT_sup, e(x'_1), …, e(x'_T), e([SEP]), e(x'_c_1), …, e(x'_c_{T_c})]`
  — the LEWIS eq.-2 concatenation of the source text `x'` and the edit template `x'_c`:
  - REPL positions: `[MASK]` in the template — but the pre-edit word is visible in the `x'` segment, so filling is a *conditioned choice* rather than unconstrained recovery (v1 hid the word entirely and the editor learned to ignore the conditioning)
  - KEEP positions: original token
  - DEL positions: **absent from the template** — deletion is the tagger's decision alone, exactly as in LEWIS where the generator never sees deleted tokens
  - insertion gaps: `[INS]` tokens occupy the slots
- **Backbone:** LLM2Vec'd Gemma + own LoRA adapter (attention/MLP projections, rank 16 default; `--lora-r 0` = frozen ablation). Embedding table stays frozen.
- **Head:** Gemma's LM head (frozen, weights tied to input embeddings); outputs token logits over the full vocabulary. The editor emits no special tokens — v2 targets are plain text — so no output-side logit correction is needed; the trainable delta rows shape only the input embeddings of `[MASK]` / `[INS]` / `[SEP]`.
- **Training:** cross-entropy over the template segment (`x' [SEP]` prefix labels are -100), with copy positions down-weighted (`--keep-loss-weight`, default 0.2)
- **Inference output:** per-position token logits over the template segment; special-token logits are suppressed before argmax. Decode: KEEP → input token, REPL/INS → argmax; detokenize.

### 4.4 Ranker

Scores each candidate output `c` produced by template enumeration:

```
score(c) =  α · sae_align(c, z_amp, z_sup)
          + β · fluency_delta(input, c)
          + γ · content_preservation(input, c)
          − η · num_INS_slots(c)
```

- `sae_align(c)`: **directional satisfaction of the commanded feature deltas**. With `Δz = pool-max(c) − pool-max(input)`, each commanded amp feature `i` contributes `clamp(Δz_i, −a_i, +a_i)` (its actual movement, capped at the commanded magnitude `a_i = z_amp[i]`) and each sup feature the mirrored term; the sum is normalised by the total commanded magnitude `Σa + Σb`, giving a magnitude-weighted score in `[−1, 1]`. The identity candidate scores exactly 0; fully realising every commanded shift scores 1; empty conditioning makes the component 0 for all candidates, so no-spec→no-edit is preserved. Magnitude weighting keeps dense (k=64) specs usable — the top features carry the signal instead of being averaged over the tail. *Why not cosine:* the original `cos(pool-max(c), z_amp) − cos(·, z_sup)` divides by the candidate's full 16k-dim norm, so thousands of unrelated active features compress the component to a ±0.005 range — ~30× too small to ever offset the content/fluency penalty of a real edit (measured on the v5 LinguaLens run: identity chosen on 100/100 pairs while 93 had edit candidates; weighted component diffs sae_align +0.002 vs content −0.021 + fluency −0.046).
- `fluency_delta(input, c)`: `min(0, tanh(meanLL(c) − meanLL(input)))` under frozen causal Gemma. A **delta**, not the absolute mean log-likelihood: sentences sit at −3..−5 nats/token, where `tanh` is saturated at ≈ −1.0 for every candidate — the v4 LinguaLens error analysis measured a 0.0003 spread between junk (`Apples apples …`) and clean candidates, i.e. the component was dead. **Clipped at 0** — a constraint, not an objective: an unclipped delta rewards fluency *gains*, and since many real inputs are stylistically marked, the ranker then edits even under **empty** conditioning (measured: empty-condition copy_rate 1.00 → 0.65 on LinguaLens at k=8), destroying the honest no-spec→no-edit property. With the clip the identity candidate (delta = 0) is unbeatable on this component; degraded candidates still score negative and `--fluency-gate` still rejects outright junk.
- `content_preservation(input, c)`: cosine of LLM2Vec sentence embeddings of input vs candidate
- Length penalty discourages over-insertion
- Default weights (tuned on dev): `α = 1.0, β = 0.3, γ = 0.2, η = 0.05`; recalibrate with `scripts/calibrate_ranker.py` on a `--dump-details` dev dump (dumps from before the fluency-delta change store saturated fluency values, and dumps from before the directional sae_align change store ±0.005-scale cosine values — neither can be reused)
- `--fluency-gate τ` (opt-in, nats/token): candidates whose meanLL drop vs the input exceeds `τ` score `−inf`. The identity candidate has delta 0 and always survives, so inference never aborts. Mirrored in `calibrate_ranker.py --fluency-gate` so offline weight search matches inference.

### 4.5 Length Predictor (ablation only)

The **main** inference path enumerates all slot configurations under `L_MAX` and selects with the ranker (the LEWIS-faithful path). The length predictor is provided as an efficiency ablation:

- Same backbone forward as the editor (`[INT_amp, INT_sup, e(editor_input)]`)
- A small MLP head reads encoder hidden states at INS positions and classifies the slot count over `{1..L_MAX}`
- **Training data:** any sample whose `tagger_gold` contains an INS run (single-op INS or compound samples that include an INS op). The gold INS-run length is read per gap from `ins_span_lengths` in the corruption cache.
- At inference, predicts a length per gap; enumeration is restricted to predicted ±1, scored by the same ranker

## 5. Vocabulary

Four special tokens exist in the vocabulary; three appear in the v2 editor input. How their embedding rows get trained depends on the LLM2Vec route:

- **In-repo route (`train_llm2vec.py`):** the specials are added to the tokenizer at the start of Stage 1 (Gemma has no native `[MASK]`), so their rows participate in MNTP from step 0.
- **McGill route (vendored, current default):** MNTP/SimCSE run on the unmodified base vocabulary; `scripts/mcgill_merge_and_expand.py` adds the specials AFTER training with mean-of-existing initialization. The rows have never seen a gradient at that point, so the editor stage trains input-side delta rows for `[MASK]` / `[INS]` / `[SEP]`. For a checkpoint expanded before v2 (no `[SEP]` yet), `scripts/add_special_tokens.py` appends missing specials idempotently — existing token ids never change, so corruption caches remain valid.

| Token | Role | Appears in editor input | Appears in editor output |
| --- | --- | --- | --- |
| `[MASK]` | placeholder at REPL positions in the template | ✓ | ✗ |
| `[INS]` | placeholder occupying an insertion slot | ✓ | ✗ |
| `[SEP]` | separates the source `x'` from the template `x'_c` | ✓ | ✗ |
| `[DEL]` | v1 legacy: output marker for deleted positions. Still stored in the corruption cache (the collator strips DEL positions using it) but absent from the v2 editor's input and output. | ✗ | ✗ |

The input format matches LEWIS's BART generator convention (`x SEP x_c`, eq. 2). Since v2 removes DEL tokens from the template — deletion being the tagger's decision — the editor's input and output are position-aligned over the template segment without any output marker.

## 6. Training Stages

### 6.0 SAE Cache — `precompute_sae.py`

Sparse top-L SAE features per LLM token over the sentence-segmented Dolma sample (see §6.2.1). Also computes `μ_f`, the per-feature mean activation, stored in `mu.npy`. Cache layout in §8.1.

### 6.1 LLM2Vec Encoder — `train_llm2vec.py`

MNTP training to turn causal Gemma into a bidirectional encoder. Follows the canonical LLM2Vec recipe (McGill-NLP) — **LoRA-based by default** (matches paper), with a `--no-use-lora` full-FT fallback for ablation:

1. **Bidirectional attention patch.** The inner `GemmaModel`'s `_update_causal_mask` is overridden to return a padding-only 4D mask (no triangular `-inf`), and every self-attention module's `is_causal` flag is set to `False`. The model is loaded with `attn_implementation="sdpa"` for speed. Gemma's SDPA path computes `is_causal = q_len > 1 and attention_mask is None and module.is_causal`; both of the latter two conjuncts are False after the patch, so SDPA falls through to bidirectional with the explicit mask. (Canonical LLM2Vec forces `eager` only because it uses a subclass-level override that registers a single attention class; we monkey-patch the existing module, so all kernels work.) Lives in `model._patch_attention_bidirectional` and is reused at inference by `BidirectionalLLM`.
2. **MNTP objective (predict-at-i-1).** Token-level masking via HF's `DataCollatorForLanguageModeling` (15% / 80-10-10), then `GemmaForCausalLM.forward(labels=labels)` whose built-in `+1` shift gives `loss = CE(logits[..., :-1, :], labels[..., 1:])`. Combined with the bidirectional attention, the masked token at position `i` is predicted from the bidirectionally-contextualized hidden state at position `i-1` — exactly the LLM2Vec objective. The LM head's pretrained "next-token" mapping is reused without retraining.
3. **LoRA adapter (canonical).** Applied AFTER the bidir patch (peft only rewires linear projections — q/k/v/o/gate/up/down — so the patched `_update_causal_mask` survives). Defaults `r=16, α=32, dropout=0.05` match the McGill-NLP `train_configs`. Training runs for ~1k optimizer steps at `LR=3e-4`. At save time `merge_and_unload()` bakes the adapter into the base, so downstream stages load a plain HF `GemmaForCausalLM` via `AutoModelForCausalLM.from_pretrained` — they remain peft-unaware. Full-FT (1e-5 / 10k steps) is also supported via `--no-use-lora`; empirically full-FT converges to MNTP loss ≈ 4.6 and STS-B Spearman 0.43 (no better than base LM), whereas the LoRA recipe reaches the paper's range.

`train_llm2vec.py` adds `[MASK]` (if absent), `[INS]`, and `[DEL]` to the tokenizer before any training step, then resizes the LLM's embedding table so MNTP trains the new rows jointly with the rest of the model (§5). The MNTP'd Gemma is the **encoder backbone** of both the editor and the tagger, and is also the **causal PPL scorer** in `corruption.py` (re-loaded with the causal mask intact, no bidir patch). It is not used as the corruption MLM (§6.2) — corruption operates at the text level and uses a separate, swappable MLM.

A standalone evaluation script `eval_llm2vec.py` (§13.3) measures (i) MNTP held-out loss, (ii) causal-PPL drift vs base LLM, (iii) bidir-vs-causal hidden-state divergence, (iv) `[MASK]` / `[INS]` / `[DEL]` embedding sanity, and (v) MTEB STS-style sentence embedding quality with pooling-strategy ablation.

### 6.1b SimCSE — `train_simcse.py`

Canonical LLM2Vec finishes with an **unsupervised contrastive** step on top of Bi+MNTP (SimCSE; Gao et al. 2021 + LLM2Vec §4.3). This is what closes the gap from ≈ 0.65 to ≈ 0.78+ on STS-Benchmark.

1. **NT-Xent loss with dropout-as-augmentation.** Each batch sentence is forwarded through the Bi+MNTP encoder **twice** with independent dropout masks; the two embeddings form a positive pair; all other in-batch sentences are negatives. Loss is the symmetric InfoNCE / NT-Xent at temperature `τ = 0.05`.
2. **Dropout injection.** Gemma-2's default `attention_dropout = 0.0` would make the two stochastic forwards identical — the loss collapses to `log B`. `train_simcse.py` injects `attention_dropout = 0.1` on every attention module before training so the two forwards differ.
3. **Pooling.** Mean over non-pad positions of the final-layer hidden state (matches `eval_llm2vec.py`'s default).
4. **LoRA on top of MNTP-merged base.** Canonical recipe: load the MNTP-merged checkpoint as a plain HF model, re-apply the bidir patch, then wrap with a **new** LoRA adapter (not the MNTP one). Defaults `r=16, α=32, LR=3e-5, batch=128, 1k steps`. At save time `merge_and_unload()` bakes the SimCSE adapter into the base — so the saved checkpoint has both MNTP and SimCSE changes baked in and downstream stays peft-unaware.
5. **Output is a drop-in `--llm2vec-dir`.** The SimCSE output dir contains a `GemmaForCausalLM` HF checkpoint (so the LM head — needed by `corruption.py` for causal PPL scoring — survives) plus an `llm2vec_meta.json` with a `simcse:` block (including `lora.merged: true`). Downstream stages (`corruption`, `tagger`, `editor`, `length_head`, `eval`) consume this dir unchanged.

`scripts/run_production.sh` runs SimCSE between stage 01 (MNTP) and stage 02 (corruption) by default. Pass `SKIP_SIMCSE=1` to skip it — useful for ablating "Bi+MNTP only" vs "Bi+MNTP+SimCSE" on downstream editor quality.

### 6.2 Corruption Data Generation — `corruption.py`

Training data is generated by applying fluency-preserving MLM-based corruption to clean sentences from Dolma. The main path is **compound corruption** (§6.2.5): each accepted sample combines N ops drawn from {REPL, INS, DEL} on a single sentence, gated by PPL and SAE-shift bounds whose thresholds scale with `sqrt(N)` and are calibrated against the empirical Dolma + MLM compound distribution (§6.2.6). Identity (N=0) and single-op (N=1) samples are retained as a small fraction so the tagger sees clean KEEP supervision and clear per-class signal.

The single-op generators in §6.2.2–6.2.4 are the building blocks of compound corruption. REPL uses the MLM to choose substitutions but always trains the editor to recover the *original* word, not the MLM's choice. INS deletes natural words at positions biased toward easy-to-remove syntactic categories (§6.2.3); no MLM recoverability gate. DEL drives word choice via the MLM (selecting plausible insertions).

#### Corruption MLM

The MLM that proposes corruption candidates is **decoupled from the downstream encoder**: it operates entirely at the text level and the corrupted text is re-tokenised with the downstream Gemma tokenizer at the end. This decoupling lets us pick any HuggingFace `AutoModelForMaskedLM` for corruption — typically a small, cloze-optimised model — while keeping the editor's Gemma encoder and LM head untouched.

The default choice is **ModernBERT-base** (~149M params, 8K context, SOTA cloze performance, ~8 ms per forward), accessible via `model.MLMProvider`. The provider exposes a text-in / text-out interface (`predict_at_masks`), so the MLM's tokenizer is encapsulated and never leaks to callers.

Other supported keys: `modernbert-large`, `deberta-v3-base`, `deberta-v3-large`, `bert-base`, `roberta-base`, `xlm-roberta-base`, or any HF model id with an `AutoModelForMaskedLM` head and a `mask_token`.

The perplexity scorer and the SAE forward remain Gemma-based; only the candidate-proposal MLM is swappable.

#### 6.2.1 Sentence-level preprocessing

Dolma documents are split into sentences before any corruption step. Each training sample is one sentence.

- Sentence splitter: any robust language-aware tool (e.g., PySBD, NLTK punkt). The splitter is the only language-dependent component of the corruption pipeline; the rest is MLM-driven and language-agnostic.
- Length filter: keep sentences with `[L_min_sent, L_max_sent]` tokens (default `[5, 256]`).
- Sentence boundaries are also used by §6.0 (`precompute_sae.py`) when building the SAE cache, so SAE features are pre-computed at sentence granularity.

#### 6.2.2 REPL data

The REPL task requires the editor to replace a token at a marked position. Training samples are generated by substituting whole WORDS in a clean sentence with MLM-sampled alternatives, then re-tokenising with the Gemma tokenizer.

```
1. Sample clean sentence X (text) from the segmented corpus.
2. Identify word boundaries in X. Choose R words to substitute.
3. For each chosen word w at character span [a, b):
   - Build masked text: X[:a] + MLM.mask_token + X[b:]
   - Run the corruption MLM → top-K predictions (decoded to text).
   - Pick a candidate w' ≠ w from {top-2..top-K} that contains a letter.
   - Substitute in the working text.
4. Re-tokenise both X (clean) and X' (corrupted) with the Gemma tokenizer.
5. Same-Gemma-token-count constraint:
   - For each substitution, the original word's Gemma-token span and the
     replacement word's Gemma-token span must have the same length, so
     that editor_input and editor_target stay 1:1 aligned by position.
   - Reject the sample otherwise.
6. Tagger gold ops:  REPL at the substituted positions in X', KEEP elsewhere.
7. Editor input:    X' with [MASK] in place of the substituted positions
                    (LEWIS-faithful).
8. Editor target:   X (the original Gemma tokens at the substituted positions).
9. Conditioning:    diff-based on (X, X') per §6.3.3.
```

The same-token-count constraint reduces yield (~20–30% of attempted REPLs are rejected on length mismatch) but keeps the editor's training signal positionally clean. The diff-based conditioning then captures the SAE features that differ between original and distractor automatically.

#### 6.2.3 INS data — word deletion

The INS task requires the editor to insert a token (or span) at a marked position. Training samples are generated by removing whole WORDS from a clean sentence and asking the editor to recover them from context + SAE conditioning — symmetric with how REPL has the editor recover the original word.

No MLM-based recoverability filter is applied. An MLM rarely predicts the EXACT natural word from context (synonyms / paraphrases score higher), so gating INS samples on MLM recoverability was over-conservative and broke symmetry with REPL/DEL, where no such gate exists. The editor learns to recover the deleted word from `z_amp` (which encodes the original's SAE features by construction) plus surrounding context.

**Position selection is biased by syntactic role.** Random position selection would generate compounds rougher than human edits (cf. §1.5, §6.2.5): the per-op MLM is conservative but the choice of which positions to touch is otherwise unconstrained. INS deletion positions are therefore biased toward syntactic categories where removal is least disruptive to grammaticality and most reflective of natural sentence variation:

- **HIGH priority** — modifiers (adjectives, adverbs), adverbial phrases (PP attached to a verb, sentential adverbs), **sentence-initial words**, function words (determiners, prepositions, conjunctions, modals, auxiliaries, possessive pronouns, `to`).
- **LOW priority** — subject / object nouns, main verbs, content words carrying core semantics.

spaCy tags each word with **Universal POS (UPOS)**. UPOS is shared across languages, so extending to a new corpus only requires swapping the spaCy model (e.g. `ja_core_news_sm` for Japanese) — the HIGH-priority tag set is unchanged. The candidate deletion span is sampled with `p_high` (default 0.85) probability from HIGH-priority spans and `1 - p_high` from LOW-priority spans. This is INS-specific. REPL position selection does not require an analogous filter because its substitution-quality gate is already implicit in the MLM top-K (low-confidence substitutions are not in the top-K so a "bad position" rarely produces a usable candidate). DEL position is MLM-driven (the MLM only proposes plausible insertions at any given site).

HIGH-priority UPOS tags: `ADJ` (adjectives), `ADV` (adverbs), `DET` (determiners), `ADP` (adpositions / prepositions), `CCONJ` (coordinating conjunctions), `SCONJ` (subordinating conjunctions), `AUX` (auxiliaries), `PART` (particles, e.g. English `to`). Plus any token at the sentence-initial position. LOW-priority: everything else, mainly `NOUN`, `PROPN`, `VERB`, `NUM`, `PRON`.

```
1. Sample clean sentence X (text) from the segmented corpus.
2. Tag word-level POS and partition word spans into HIGH / LOW priority
   per the rules above. Sentence-initial word is always HIGH.
3. With probability p_high (default 0.85), sample a candidate deletion
   span from HIGH-priority words; otherwise from LOW. Choose n ∈
   {1..N_MAX_INS} consecutive words within the selected pool, leaving
   at least one word on each side of the deletion.
4. X' = X with the chosen words removed (one trailing whitespace absorbed).
5. SAE-shift filter:
   z_X  = sae(...)
   z_X' = sae(...)
   reject if ||z_X - z_X'|| outside sae_min(N) .. sae_max(N) (§6.2.6).
6. Fluency filter:
   reject if ppl(X')/ppl(X) > ppl_max(N) (§6.2.6).

7. Re-tokenise X and X' with the Gemma tokenizer.
   - The deleted span's Gemma token count L = len(x_ids) - len(xp_ids),
     derived from the token-count diff (robust to SentencePiece leading-
     space encoding). This is the **gold INS span length** consumed by the
     length predictor (§4.5).
   - Verify a clean prefix/suffix transposition: the longest common
     prefix between x_ids and xp_ids identifies the gap position; the
     suffix after the deleted span must match. Reject otherwise.

Training tuple (single INS op, before compound composition):
   tagger input  = X'   (Gemma tokens)
   tagger gold   = INS at the gap position in X'
                    (single op per gap; the L slot count is taken from
                     `ins_span_lengths` in the cache record)
   editor input  = X' with L consecutive [INS] tokens inserted at the gap
   editor target = X    (Gemma tokens; aligned with editor input by construction)
   conditioning  = diff-based on (X, X'), z_amp from positive delta
   gold_span_length = L

When this op participates in a compound (§6.2.5), the same (gap, L)
contribution is composed with the gap/length contributions of the
other ops on the same sentence; `ins_span_lengths` then records one
entry per INS gap in the final tagger gold.
```

Multi-word spans naturally produce length-supervision for the length predictor.

#### 6.2.4 DEL data — word insertion under MLM plausibility

The DEL task requires the editor to remove an unwanted token (or span) at a marked position. Training samples are generated by inserting whole WORDS at positions where the MLM predicts them to fit, then asking the editor to delete them.

```
1. Sample clean sentence X (text) from the segmented corpus.
2. Choose an insertion position p (after a randomly chosen word) and a
   word-span count n ∈ {1..N_MAX_DEL}.
3. Insert one word at a time, top-1 / top-K mix:
       for k in range(n):
           masked = current[:p] + " " + mask_token + current[p:]
           preds  = MLM(masked).top_K  # text-level decoded candidates
           w = top-1 (with prob TOP1_PROB) else sample from top-{2..K}
           current = current[:p] + " " + w + current[p:]
   Final text X+γ is X with γ inserted at p.
4. SAE-shift filter:
   z_X       = sae(...)
   z_{X+γ}   = sae(...)
   reject if ||z_X - z_{X+γ}|| outside sae_min(N) .. sae_max(N) (§6.2.6).
5. Fluency filter:
   reject if ppl(X+γ)/ppl(X) > ppl_max(N) (§6.2.6).

6. Re-tokenise X+γ with the Gemma tokenizer. Locate the inserted γ in the
   token sequence by character offsets; record its Gemma-token length
   `del_span_length`.

Training tuple:
   tagger input  = X+γ (Gemma tokens)
   tagger gold   = DEL at each Gemma position covering γ
   editor input  = X+γ                       (garbage in place; no [MASK])
   editor target = X+γ with [DEL] at each γ position
                   (KEEP target identity elsewhere)
   conditioning  = diff-based on (X, X+γ), z_sup from positive (-delta)
```

Position selection is MLM-driven rather than rule-based: any position where the MLM predicts a plausible word is a valid insertion site. This keeps the pipeline language-agnostic beyond sentence segmentation.

#### 6.2.5 Compound corruption — unified op sampling

Compound corruption layers multiple ops on a single clean sentence by composing the single-op generators of §6.2.2–6.2.4. This is the main path for training data and the dominant bucket in §6.3.1. A single accepted sample is constructed as follows:

```
1. Sample clean sentence X.
2. Sample N ~ truncated geometric (p ≈ 0.4), capped at N_MAX (default 5).
   N=0 is identity. N=1 is single-op. N ≥ 2 is multi-op compound.
3. Sample op types op_1..op_N i.i.d. from {REPL, INS, DEL} with the
   PROPOSAL weights (default 0.60, 0.34, 0.06). These are pre-gate
   sampling weights, calibrated so the ACCEPTED op-type mix matches the
   LinguaLens English ratio (REPL 0.70 / INS 0.18 / DEL 0.12;
   `runs/lingualens_token_diff.md`) after the non-op-type-neutral gates
   of §6.2.6 — see §6.2.7.
4. Assign each op a non-overlapping word span on X. INS spans are drawn
   with the priority bias of §6.2.3. REPL and DEL spans use uniform
   random word positions (REPL relies on MLM top-K for substitution
   quality; DEL position is implicitly filtered by the per-op MLM
   plausibility check).
5. Apply ops right-to-left so that earlier ops' position bookkeeping is
   not invalidated by length-changing later ops. The result is the
   compound corrupted text X'.
6. Re-tokenise X' with the Gemma tokenizer.
7. Gate the compound by the N-dependent PPL and SAE-shift thresholds of
   §6.2.6. If the compound passes, record it as one sample with the
   union of all op-gold positions (per-token 4-class tagger gold over
   KEEP / REPL / INS / DEL).
8. Otherwise discard and re-try (first-accept rejection sampling within
   K_BUDGET attempts per source sentence, default 6). After K_BUDGET
   failures the source sentence is skipped.
```

The first-accept policy preserves the distribution conditional on passing the gate. We deliberately do **not** rank K candidates by PPL: such a "smoothest-is-best" ranking would push the editor's training distribution toward the MLM's most conservative compounds. We want the editor to see the full natural distribution of compounds that pass the gate, not just the safest fraction.

Per-position op composition is handled by `apply_ops_for_editor` and the tagger gold construction, both of which already operate on per-position op labels — no architectural change is required downstream.

#### 6.2.6 N-dependent gates and self-consistent calibration

Two independent gates are applied to each (X, X') pair:

```
fluency  :  slor_drop = SLOR(X) − SLOR(X')                 <= slor_drop_per_op * N    (linear)
SAE      :  |top_K_local(X) \ top_K_local(X')|              >= sae_min_topk_change    (binary by default)

SLOR(s)        = (1/|s|) * [log p_M(s) − log p_unigram(s)]
top_K_local(z) = the set of K SAE features with the highest pool-max activation
                 RESTRICTED to tokens that overlap an edited char range.
```

Fluency uses **SLOR** (Pauls&Klein 2012 / Lau et al. 2017 / Kann et al. NAACL 2018) — per-token, unigram-normalised log-likelihood — instead of raw PPL ratio, which has well-documented length and rare-word artefacts (Wang+ 2022, Kann+ 2018).

The **SAE gate** is a **top-K identity change check** (BINARY by default): reject the pair if the top-`K` most-active SAE features at the *edited* token positions are the same SET before and after corruption. The L2 shift was the previous metric but conflates magnitude wiggles with actual representation change — if the top-K features have the same identities, the editor cannot learn from the conditioning vector even when `||z_X − z_X'||` is non-zero. The default `sae_min_topk_change = 1` means "at least 1 feature must change identity"; raise it for stricter SAE-space separation.

**Why local pool-max instead of global.** The first empirical sweep used pool-max over ALL token positions and saw `top_K change p50 = 0` for every `N ∈ {1..5}` at K=10, with corresponding yield ~0% — even though L2 shift grew from 1.5 to 11.8 across the same N range. The diagnosis: the global top-K is dominated by **content-independent features** (language id / BOS-EOS / punctuation / sentence-initial position) whose pool-max happens at unedited positions, so local edits leave the global top-K identity unchanged even when they substantially move the SAE representation. Restricting the pool to tokens that overlap an edited char range removes that floor: the gate measures whether the SAE features *where the edit actually landed* changed. The global top-K is still recorded in calibration JSONL as `sae_topk_change_global` for diagnostic comparison.

**No L2 upper bound.** An earlier design included a `sae_max = sae_per_op_max * sqrt(N)` L2 cap to enforce minimality. It was removed: top-K identity change already captures whether the SAE-space interpretation moved, and a corruption that flips a small number of top-K features with high magnitude — exactly the case the editor learns most from — would be wrongly rejected by an L2 cap. The raw L2 shift is still recorded in calibration / telemetry for offline analysis but is not gated against.

**Why SLOR.** Raw `PPL(X')/PPL(X)` over-penalises corruption that inserts a rare-but-fluent word (because the rare word's marginal probability is low) and conflates fluency change with length change (because PPL is a function of length). SLOR subtracts a unigram baseline and divides by `|s|`, which structurally removes both artefacts. Kann+ 2018 report a Pearson correlation of `0.454` for SLOR vs `0.325` for raw PPL on referenceless fluency evaluation. The same paper's "of Tuvalu" vs "of France" example mirrors what happens in our pipeline when the corruption MLM proposes rare-but-natural words: under raw PPL the candidate is wrongly rejected; under SLOR it survives.

**Why linear N for SLOR drop.** Per-token Δlog-likelihood per op is approximately constant under the empirical fit (we observed `factor^N` matching p50 of raw PPL ratio for `N ∈ 1..5` with `factor = 1.25 ≈ exp(0.22 nats/token/op)`), which says the expected SLOR drop is `(constant) × N` — linear, not `sqrt(N)`. SAE-shift retains `sqrt(N)` (a separate decision, calibrated independently against the same data).

**Calibration.** The fluency constant is not fixed by theory; it is fit by running the compound generator on Dolma in a **calibration mode** that records every attempt's `(N, slor_drop, sae_shift, sae_topk_change, plus legacy ppl_ratio)` without applying the gate. For each `N ∈ {1..N_MAX}`, percentile statistics drive the threshold:

| Constant | Setting |
| --- | --- |
| `slor_drop_per_op`     | such that `slor_drop_max(N)` ≥ 75th percentile of `slor_drop` at each N |
| `sae_min_topk_change`  | not a percentile — set per design (`1` = strict binary; raise for stricter separation) |

The calibration is **self-consistent on the same Dolma + MLM compound distribution that the editor will train on**: the calibration source = the training source. We do not calibrate against external paraphrase corpora — paraphrase pairs have a fundamentally different position distribution (humans target meaning-relevant positions; this pipeline uses priority-biased random positions), so the metrics are not directly comparable and would mis-fit the threshold.

**Unigram baseline.** Built once at startup from `--unigram-sample-size` Dolma sentences using the downstream Gemma tokenizer (skipping `<bos>` / `<eos>` / `[PAD]` / `[MASK]` / `[INS]` / `[DEL]`), with add-`unigram_smoothing` Laplace over the observed vocabulary plus one UNK bucket. Cached to `<out-dir>/unigram.json` so re-runs reuse the same baseline.

The expected per-attempt yield after calibration is ~50% at the chosen percentiles, keeping `K_BUDGET = 6` adequate for almost all source sentences.

#### 6.2.7 Op-type proposal weights — gate-aware calibration

The op-type ratio that matters is the one the editor and tagger actually train on: the **accepted** distribution, after the rejection-sampling gates of §6.2.6. Our target for that accepted distribution is the LinguaLens English op-type histogram (REPL 70.0% / INS 18.2% / DEL 11.8%; `runs/lingualens_token_diff.md`).

The gates are **not op-type-neutral**, so sampling op *proposals* directly at the target ratio does not reproduce it after gating. The dominant gate is the SLOR fluency bound, and the three op types load it very differently *by construction*:

- **INS** deletes real words — modifiers, function words, sentence-initial tokens (§6.2.3) — with **no MLM recoverability gate**: the span is chosen for syntactic role, not for how fluent the residue reads. Its X′ therefore incurs the largest average SLOR drop and is rejected most often.
- **DEL** inserts words the MLM judges **plausible at that position** (§6.2.4), so X′ stays fluent by construction and is rejected least often.
- **REPL** substitutes via MLM top-K (§6.2.2), a moderate fluency perturbation, and is accepted at close to its proposal rate.

Concretely, with op gold-alignment correct, sampling proposals at the target weights `p⁰ = (REPL 0.70, INS 0.18, DEL 0.12)` yields an accepted mix of `q ≈ (REPL 0.69, INS 0.06, DEL 0.25)` measured over ~2.7k accepted ops on a pilot shard — INS undershoots the target by ≈3× and DEL overshoots by ≈2×, exactly matching the mechanism above.

We therefore treat the proposal weights as free parameters and calibrate them so the **accepted** mix matches the target. From the pilot, define the effective per-op acceptance factor

```
a_op = q_op / p⁰_op          →   a_REPL ≈ 0.98,  a_INS ≈ 0.33,  a_DEL ≈ 2.09
```

and set the proposal weights to the damped inverse-acceptance reweighting of the target `t = (0.70, 0.18, 0.12)`:

```
w_op ∝ t_op / a_op^γ ,   renormalised to sum 1,     γ ∈ (0, 1]
```

The damping exponent `γ` trades exactness against robustness. Full inversion (`γ = 1`) drives the accepted mix onto the target *in expectation* but overfits a single pilot shard's finite-sample acceptance estimate (and would swing DEL down to ≈0.04, where ordinary sampling noise across shards becomes large relative to the weight). We use `γ ≈ 0.65`, giving

```
w = (REPL 0.60, INS 0.34, DEL 0.06)
```

which is the default in `corruption.py` and the `OP_WEIGHT_{REPL,INS,DEL}` default in `scripts/corruption_parallel.sh`.

**Measured outcome — proposal reweighting saturates.** The full production run (100k samples, 133,350 accepted ops) under `w = (0.60, 0.34, 0.06)` yields an accepted mix of

```
q = (REPL 0.705, INS 0.074, DEL 0.221)
```

Near-doubling the INS proposal weight (0.18 → 0.34) moved the accepted INS share by only +1.4 points (0.060 → 0.074), and halving the DEL weight barely moved DEL (0.251 → 0.221). The two operating points give a log–log sensitivity of accepted-INS share to its proposal weight of ≈ 0.33; at that slope, reaching the 0.18 target would require `w_INS > 1`, which is infeasible. The mechanism is the `K_BUDGET` first-accept loop (§6.3.2): a rejected INS proposal releases its attempt slot to the next draw — usually an easily-accepted REPL or DEL — so the accepted mix is governed by the per-op gate pass rates far more than by the proposal weights. Equivalently, the acceptance factors `a_op` are not constants: they fall as `w_op` rises (measured `a_INS`: 0.33 → 0.22 when `w_INS` went 0.18 → 0.34), making the correction self-limiting. Proposal reweighting can therefore only *partially* close a gate-induced gap.

We keep `w = (0.60, 0.34, 0.06)` as the operating point — it maximises the INS share within the feasible range without starving DEL — and report the accepted distribution as measured rather than claiming a match to the LinguaLens target. The residual gap (INS 0.074 vs 0.18, DEL 0.221 vs 0.12) is a property of the fluency gate, not of the sampler: closing it would require changing the data-generating process itself, e.g. an op-conditional SLOR allowance for INS (defensible, since deleting a natural word lowers fluency *by construction*, so a uniform threshold implicitly penalises the op type rather than the sample quality) or a fluency-aware INS span proposer. We leave that as an explicit design decision rather than folding it silently into the weights; class-weighted losses in Phase A (§6.3) absorb the remaining imbalance on the training side. Because the acceptance factors depend on the gate threshold, the corruption MLM, and the SAE/encoder, the accepted mix is **re-measured on the first shards of any run** after a configuration change — the same self-consistency principle as the SLOR calibration in §6.2.6.

This calibration is orthogonal to the two other distributional controls: §6.2.6 sets *how many* compounds pass at each N (per-N yield), §6.3.1 sets the *N* distribution (op-count per sample), and this step sets the op-*type* distribution within accepted compounds. All three are calibrated on the same Dolma + MLM source the editor trains on.

#### 6.2.8 Condition-selective records — making the conditioning necessary

Held-out probes on the v1/v2 caches showed the editor scoring conditioning-IGNORED (and end-to-end LinguaLens editing moving the SAE features nowhere, `sae_shift ≈ 0`) even after the LoRA-capacity and `x' [SEP] x'_c`-visibility fixes. The root cause is informational, not architectural: **the denoising task is solvable without the conditioning.** Given the corrupted `X'`, the clean `X` is essentially determined by context, so `z_amp`/`z_sup` are redundant and the loss can be minimised while ignoring them. Two secondary defects compound this: the empty-conditioning hack zeroed `z` while KEEPING the full-restore target — explicitly teaching "edit without being asked" — and the cache contained no example of a corrupted-looking sentence whose gold is all-KEEP, so the tagger learned the marginal prior "~95% of inputs need editing" (and duly edited 90% of out-of-domain LinguaLens inputs, driving `sim_target` below the input-copy baseline).

The v3 cache makes the conditioning **necessary by construction**. From every accepted compound with ops `{o_1..o_N}`, up to three records are emitted (probabilities `--emit-full/-partial/-null`, defaults 1.0 / 1.0 / 0.5):

| kind | clean side | gold | conditioning |
| --- | --- | --- | --- |
| `full` | `X` (all ops reverted) | all ops tagged | `diff(SAE(X), SAE(X'))` — as before |
| `partial` (N ≥ 2) | `X_S` — a random proper nonempty subset `S` reverted | ONLY `S` tagged; the complement's corruptions are gold-KEEP | `diff(SAE(X_S), SAE(X'))` |
| `null` | `X'` itself (`S = ∅`) | all-KEEP | identically zero (`diff(X', X') = 0`) |

The same `X'` therefore appears with *different* gold depending on `S`, and the conditioning is the only disambiguator — a model that ignores `z` cannot fit the data. The `null` records simultaneously teach "corrupted-looking text + no instruction → edit nothing" with the *correct* target, replacing the empty-conditioning hack (retired: `--empty-cond-prob 0.0`). Training and inference are aligned for the first time: at inference the user names features, and the model performs exactly the edits those features select.

Implementation: subsets are rebased from `X` onto `X_S` at the character level (`rebase_subset_ops`; the non-overlap claim rules guarantee each subset op's span survives verbatim in `X_S`, which the code verifies by substring equality — composition `apply(rebase(S), X_S) = X'` is checked exactly in tests), then the ordinary `build_compound_sample` machinery runs on `(X_S, S)` unchanged. Partial/null records skip the SLOR/PPL gates (`finalize_sample(light=True)`) — the parent compound already passed them — and cost only the extra SAE forward for `X_S` (zero extra forwards for `null`). Records carry `subset_kind` / `n_parent` metadata; the record schema is otherwise identical, so the collator, training, and evaluation code consume a v3 cache unchanged.

#### 6.2.10 Linguistically-typed transformation ops (v4) and confound-controlled conditioning

The ceiling measurement (gold-in-top-k) showed the lexical-restore ops hitting a structural ambiguity ceiling (top-50 ≈ 0.52–0.59): corrupted context plus a few SAE features underdetermine the exact original word. v4 adds **rule-based grammatical↔grammatical transformation ops** (`transforms.py`, indexed on UniMorph/UD dimensions, spaCy + lemminflect) whose gold is near-deterministic given the feature — TENSE / ASPECT / MODALITY (+NPI-aware NEGATION) / NUMBER+agreement / DEGREE / DET-QUANT / ANAPHOR, and the structural centrepiece: VOICE, INTERROG (Y/N, wh; island constraints are encoded as applicability restrictions — only matrix arguments of mono-clausal sentences are ever extracted), EXISTENTIAL, VALENCY, TOUGH, ELLIPSIS, CLEFT, INVERSION, SPLIT/JOIN. BLiMP's 12 phenomenon categories map onto this inventory as ops (alternations), coordination requirements (agreement), applicability constraints (islands) or item-swap sub-rules (NPI); BLiMP data itself is never trained on.

Quality is enforced in three layers: conservative dependency-pattern applicability; a **round-trip check** (a reverse-direction proposal on T(X) must reproduce X verbatim) for invertible families; and a **symmetric SLOR gate** (`--transform-slor-delta`): both sides are supposed to be grammatical, so the gate bounds |SLOR(X) − SLOR(T(X))| instead of the one-sided corruption drop. Gold for arbitrary pairs (reordering included) is built by token-level diff (`pair_to_gold`), which subsumes the OpSpec machinery for these records. **Both directions are emitted** (X→T(X) and T(X)→X): the same surface forms appear with opposite conditioning and opposite gold, so the edit direction is decidable only through z.

Conditioning is extracted **edit-locally** (`--cond-scope local`, default): the stored top-k comes from pool-max over the tokens overlapping the edited char ranges, not the whole sentence, raising its information content about the edit. Before the top-k, the **generic-grammaticality blocklist** (`--blocklist`, §scripts/build_grammaticality_blocklist.py — features responding across ≥20 unrelated BLiMP paradigms with consistent sign) is masked out, so conditioning cannot degenerate into a "change grammaticality" axis; the pre-mask blocked mass is recorded per record (`cond_blocked_mass`) and aggregated per transformation type in `meta.json` as a standing diagnostic.

**v4d families — edit-geometry variation (v5 data).** The LinguaLens error analysis showed the pipeline generalises end-to-end to phenomena *inside* the training inventory (interrogative inversion transferred with sim = 1.0) and fails on operations outside it — so v5 extends the inventory along the **edit-geometry axes the first 17 families under-covered**, not toward any particular benchmark: pure reordering with unchanged material (PARTICLE `pick up the book ↔ pick the book up`, DATIVE `give the book to Mary ↔ give Mary the book`, ADVPLACE pre-verbal ↔ clause-final `-ly` adverbs, PPFRONT sentence-final ↔ fronted adjunct PPs, QUOTINV `," said John ↔ ," John said`), token fusion/split (CONTRACT `do not ↔ don't`), and clause-edge function words (COMPZR bridge-verb `that`-omission, NONFIN `began to read ↔ began reading`). All 25 families ride the same defence stack (conservative dependency predicates → round-trip → symmetric SLOR → blocklist).

Two generalization mechanisms ship with them:
- **Composition** (`--transform-compose-prob`, default 0.15): an accepted T1(X) is chained with a second family T2 (re-parsed; per-step round-trip; the symmetric SLOR gate sees the composed pair), emitted as `t_type "A+B"`. Unseen edits are often unseen *combinations* of seen operations; composed records teach the compositional semantics of feature specs directly.
- **LOFO (leave-one-family-out)**: every transform record now stores `t_family`; `CorruptionDataset` filters by it at LOAD time (`--exclude-families` on the train scripts, `--only-families` on `eval_tagger_editor.py`), so a LOFO experiment reuses one shared corruption cache — no regeneration, one extra reduced-scale training pair (exclude vs. control) measures generalization to held-out operation types. Pre-v5 records without `t_family` are mapped from the `t_type` prefix.

### 6.3 Phase A — Unified Corruption Pretraining (`train_editor_phaseA.py`)

A single end-to-end training phase trains the editor and the tagger on the same corruption stream.

#### 6.3.1 Sample distribution

| Bucket | Probability | Source |
| --- | --- | --- |
| Identity (N=0) | 5% | clean Dolma sentence, empty conditioning; KEEP-only tagger gold |
| Single-op (N=1) | 55% | one of REPL / INS / DEL with the bucket-N=1 weight; clean tagger op-discrimination |
| Compound N=2..3 | 35% | §6.2.5 with N drawn from the truncated geometric restricted to {2, 3} |
| Compound N=4..N_MAX | 5% | §6.2.5 with N ≥ 4 |

These weights are calibrated against the **LinguaLens English minimal-pair `n_hunks` distribution** (see `scripts/lingualens_token_diff.py` and `runs/lingualens_token_diff.md`), which finds 66.6% N=1 / 28.9% N=2 / 3.6% N=3 / 0.1% N≥4 — minimal pairs in the wild are dominated by single substitutions and short two-op compounds, not by deep compounds. We bias slightly more toward N=2-3 than the empirical 32.5% (35%) and retain a 5% N≥4 long-tail bucket so the editor still sees occasional harder cases. The earlier (10 / 15 / 45 / 30) split, which assumed compositionality was the dominant inference-time regime, over-trained on compound N=4+ at ~300× the empirical rate.

Bucket selection is implemented as **rejection on N after sampling**: a fresh sentence is run through the compound generator, the realised N determines the bucket, and the sample is accepted into the stream if the bucket's per-N quota has not yet been filled. Per-bucket and per-N accept rates are logged in `meta.json`.

#### 6.3.2 Sampling parameters

| Variable | Range | Distribution |
| --- | --- | --- |
| `N_total` per sample | 0..N_MAX (default 5) | truncated geometric, p ≈ 0.4 |
| op type within compound (proposal) | {REPL, INS, DEL} | (0.60, 0.34, 0.06) — pre-gate weights; accepted mix measured 0.705 / 0.074 / 0.221, gate-limited (§6.2.7) |
| `p_high` (INS HIGH-priority bias) | 0.85 | flat |
| `ins_span_length` per INS op | 1..L_MAX | log-uniform |
| `del_span_length` per DEL op | 1..L_MAX_DEL | log-uniform (typically L_MAX_DEL ≤ L_MAX) |
| `K_BUDGET` (first-accept attempts per source sentence) | 6 | budget |
| `K_train` (top-K for SAE pool max) | 64 | fixed |
| `K_top` (positive / negative diff candidates) | 8 | fixed |
| `k_amp` (per sample) | {1, 2, 3, 4} | uniform |
| `k_sup` (per sample) | {1, 2, 3, 4} | uniform |
| Empty conditioning probability | 0.0 — retired; §6.2.8 null records carry zero-conditioning supervision with correct targets | — |

#### 6.3.3 Conditioning vectors

For every accepted sample `(X, X')`, the conditioning is constructed automatically from the SAE forward of each text:

```python
z_X       = sae_pool_max_topk(encode(X),       K=K_train)
z_X_prime = sae_pool_max_topk(encode(X_prime), K=K_train)
delta = z_X - z_X_prime

amp_candidates = topk_indices(max(delta, 0),  K_top)
sup_candidates = topk_indices(max(-delta, 0), K_top)

# Conditioning sub-sampling
k_amp = random.choice([0, 1, 2, 3])
k_sup = random.choice([0, 1, 2, 3])
amp_idx = sample(amp_candidates, k_amp)
sup_idx = sample(sup_candidates, k_sup)

z_amp = sparse(amp_idx, delta[amp_idx])
z_sup = sparse(sup_idx, -delta[sup_idx])

if random() < empty_conditioning_prob:    # 0.05 default
    z_amp = z_sup = sparse_zeros()
```

This routes the SAE difference between corrupted and clean text into conditioning vectors that match the SAE semantics by construction. The small per-sample `k_amp` / `k_sup` and the empty-conditioning samples together force the model to learn the conditioning interface as a feature-by-feature signal rather than as a low-entropy index of the target.

To accelerate training, `z_X` and `z_X_prime` are precomputed once per sample during `corruption.py` and stored in the corruption cache (§8.2). The training loop only performs the sub-sampling.

#### 6.3.4 Input / target construction

The corruption CACHE stores the v1 artifacts (4-class `tagger_gold`; a template with in-place DEL tokens and `[DEL]` targets). `data.CorruptionCollator` derives the v2 form at batch time — **no cache regeneration is needed**:

```
cache record                      v2 batch (derived by the collator)
──────────────────────────        ─────────────────────────────────────────────
tagger_gold (4-class,             tagger_op3_gold ∈ {KEEP, REPL, DEL}
  INS on gap-adjacent token)  →   tagger_ins_gold ∈ {0, 1}   ("insert before")
                                  (split_cache_tags: a cached INS tag always
                                   sits on an otherwise-KEEP token)

editor_input / editor_target  →   editor input:  x' [SEP] x'_c   where x'_c =
  (v1 template with DEL kept        cached template with DEL positions
   in place, target [DEL])          (target == [DEL]) removed and the
                                    duplicated leading <bos> dropped
                                  editor target: -100 over the x' [SEP]
                                    prefix; clean-reference tokens over x'_c
                                    (REPL → original word, [INS] slots →
                                     inserted words, KEEP → identity)
```

A single insert indicator opens one `[INS]` gap; the gold slot count per gap is `ins_span_lengths` from the cache (template enumeration varies it at inference, §11).

#### 6.3.5 Loss and trainable parameters

- **Editor loss:** cross-entropy over the template segment (the `x' [SEP]` prefix is -100), with copy positions down-weighted by `--keep-loss-weight` (default 0.2) so the gradient concentrates on edit positions:
  - KEEP positions enforce identity at weight 0.2 (suppresses over-editing)
  - REPL positions enforce restoration at weight 1 (trains amplification via `z_amp`, suppression of distractor features via `z_sup`) — with the pre-edit word visible in the `x'` segment
  - INS positions enforce filling at weight 1 (trains amplification via `z_amp`)
  - (no DEL positions: they never enter the template)
- **Tagger loss:** class-weighted cross-entropy (3-class op head) + BCE with capped pos_weight (binary insert head)
- **Trainable:** `Proj_A`, `type_emb[0..2]`, `cond_scale`, tagger op/ins heads, input-side delta rows for `[MASK]` / `[INS]` / `[SEP]`, and a per-model backbone LoRA adapter (rank 16 on attention/MLP projections, fp32, its own LR group `--backbone-lr 1e-4`; `--lora-r 0` = frozen ablation)
- **Frozen:** LLM2Vec base weights, SAE, LM head, original input embeddings

Editor warmup: `Proj_A` is frozen for the first ~5% of steps (only `[MASK]` / `[INS]` / `[SEP]` deltas, `cond_scale` and `type_emb` update), then unfrozen. The tagger trains after the editor, warm-starting its conditioning interface from `editor-final.pt`, with the same initial `Proj_A` freeze.

### 6.4 Length Predictor (ablation) — `train_length_head.py`

Trained on any sample whose `tagger_gold` contains an INS run. The gold span length per gap is read from `ins_span_lengths` in the corruption cache. Backbone and `Proj_A` are frozen; only the length MLP head is trained.

## 7. Inference (`evaluate_intervention.py`)

```python
def edit(text, intervention_spec, strength=1.0):
    z_amp, z_sup = build_intervention_vectors(intervention_spec, mu, strength)
    INT_amp = Proj_A(z_amp) + type_emb[1]
    INT_sup = Proj_A(z_sup) + type_emb[2]

    # 1. tagger — two tags per token (LEWIS §2.1)
    text_ids = tokenizer(text).input_ids
    op3, ins_before = tagger.predict_ops(text_ids, INT_amp, INT_sup)

    # 2. assemble the edit template x'_c and identify INS gaps
    #    KEEP → original token, REPL → [MASK],
    #    DEL → token REMOVED, ins_before → one [INS] slot per gap
    template = apply_ops_for_editor(text_ids, op3, ins_before)

    # 3. enumerate slot configurations (main path)
    candidates = []
    for slot_counts in product(range(1, L_MAX + 1), repeat=len(template.ins_gaps)):
        cur = expand_slots(template, slot_counts)
        # LEWIS eq. 2: the editor sees both the source and the template
        logits = editor(text_ids + [SEP] + cur.input_ids, INT_amp, INT_sup)
        tpl_logits = logits[len(text_ids) + 1:]         # template segment
        tpl_logits[:, special_token_ids] = -inf         # never emit markers
        output = decode_editor_output(cur.input_ids, argmax(tpl_logits), cur.op_per_pos)
        # decode: KEEP → input token, REPL/INS → argmax
        # (deletion already happened: DEL tokens are not in the template)
        candidates.append(output)

    # 4. rank candidates
    scored = [(c, ranker.score(c, text, z_amp, z_sup)) for c in candidates]
    best = max(scored, key=lambda x: x[1])[0]
    return best
```

**Candidate diversity (LEWIS's beam analog).** One tagger forward yields several *edit plans* at increasing strictness (`--op-thresholds`, default `0.0,0.9`: plain argmax, and a plan keeping REPL/DEL decisions only at softmax probability ≥ 0.9 with the insert bar raised to match), and the **unedited input is always in the candidate pool**, so the ranker can overrule a trigger-happy tagger instead of being forced to pick among its edits. Plans whose slot enumeration would exceed `--max-templates` are dropped; the identity candidate always survives, so inference never aborts.

**Iterative refinement (`--refine-passes`, default 1 = off).** Coordinated multi-site edits — voice flip, subject-verb inversion — exceed what a single pass can express: fill refinement varies **one** position at a time, so the candidate pool only ever contains partially-moved intermediates (v4 LinguaLens error analysis: `The game was won by the team` → best candidate `The team was won by the team`, subject swapped but object not yet). `edit_iterative()` re-runs the whole tagger → enumeration → ranker loop on the output under the **same** `(z_amp, z_sup)` spec: the features the spec names still mismatch at the remaining sites, so the tagger fires there on the next pass. It stops early at a fixpoint (the identity candidate guarantees one exists), so `passes` is a cap, not a target. `eval_lingualens.py` additionally offers `--refine-recompute` (true condition only): each pass re-derives the spec as `diff(z(current), z(target))` — the remaining gap to the same user-supplied target feature state — so already-achieved features stop driving edits.

The length-predictor ablation replaces step 3 with a single forward through the length head plus `±1` enumeration around the prediction, scored by the same ranker.

## 11. Multi-Token Insertion

- A single insert indicator (`ins_before = 1`) produces exactly one `[INS]` gap.
- The slot count per gap is **not** decided by the editor; it is fixed by the input template.
- The main inference path follows LEWIS: enumerate `{1..L_MAX}` slot counts per gap, run the editor on each template, score with the ranker, take the top-1. With `L_MAX = 5` and typical `G ≤ 3` INS gaps per edit, this is bounded at `L_MAX^G ≤ 125` editor forwards per example, batched in parallel.
- For deployment-cost scenarios, the length predictor reduces this to `1 + 3^G` forwards with negligible quality loss (reported as an ablation).
- Training-time supervision for INS slot counts comes from INS ops generated by §6.2.3 (both within compound samples and pure single-op samples). The gold length per INS gap equals the Gemma-token length of the deleted word span recorded by the corruption generator.
- Raise `L_MAX` for languages or domains with longer insertion patterns.

## 12. Implementation Order

1. `lewis_ops.py`: two-tag op semantics and template construction (REPL → `[MASK]`, DEL → removed, insert indicators → `[INS]` gaps)
2. Vocabulary expansion: add `[MASK]` (if missing), `[INS]`, `[DEL]`, `[SEP]` at the start of `train_llm2vec.py` (or via `scripts/add_special_tokens.py` for an existing merged checkpoint); resize tokenizer + LLM2Vec checkpoint embedding table
3. Sentence segmentation utility
4. `corruption.py`:
   a. Single-op building blocks: REPL substitution, INS priority-biased deletion (POS tagger integration), DEL MLM-plausible insertion
   b. Compound corruption (§6.2.5): N sampling, non-overlapping position assignment, right-to-left op application, first-accept rejection with `K_BUDGET`
   c. N-dependent gates with `sqrt(N)` scaling (§6.2.6); calibration mode that records every attempt's `(N, ppl_ratio, sae_shift)` for percentile fitting
   d. Precompute `z_X` and `z_X_prime` top-K_train into the corruption cache
   e. Sample-bucket sampler over identity / single-op / compound-by-N; per-bucket and per-N yield logging
5. `tagger.py` and `train_tagger.py`: 3-class op head + binary insert head, class-weighted CE + pos-weighted BCE on corruption-derived ops, diff-based conditioning sub-sampler
6. `editor.py`: bidirectional encoder forward over `x' [SEP] x'_c` + frozen LM head wiring
7. `train_editor_phaseA.py`: unified corruption denoising with diff-based conditioning, keep-weighted CE, warmup schedule
8. `ranker.py`: SAE alignment + fluency + content + length penalty
9. `evaluate_intervention.py`: tagger → editor enumeration → ranker pipeline + metrics
10. (Ablation) `length_head.py`, `train_length_head.py`: efficiency variant
11. End-to-end evaluation; tune ranker weights on dev set

## 13. Evaluation

### 13.1 Per-stage metrics

| Metric | Stage | Purpose |
| --- | --- | --- |
| op-level accuracy | tagger (op head) | per-class precision / recall over KEEP / REPL / DEL |
| insert P/R/F1 | tagger (ins head) | binary "insert before this token" detection |
| REPL accuracy | editor | argmax = ground-truth target at template `[MASK]` positions |
| INS accuracy | editor | argmax = ground-truth target at template `[INS]` slots |
| KEEP accuracy | editor | argmax = original token at template KEEP positions |
| marker emission rate | editor | any special token winning an argmax in the template (always wrong in v2) |
| edit distance | end-to-end | Levenshtein vs reference (where available) |
| BLEU / chrF / SARI | end-to-end | LEWIS-comparable text generation metrics |
| span IoU | end-to-end | union of both tagger heads' edit positions vs gold |
| flip rate | end-to-end | classifier flip from source to target style (where labelled) |

### 13.2 Ablations

| Ablation | Purpose |
| --- | --- |
| `no-sae-shift-filter` | Disable the SAE-shift filter on both INS and DEL. Tests whether SAE-aligned conditioning relies on samples with measurable feature shift. |
| `no-fluency-filter` | Disable the perplexity filter on both INS and DEL. Tests the contribution of input / target fluency to downstream performance. |
| `no-subsample` | Use all top-K_top features as conditioning, with no per-sample sub-sampling. Tests the information-leak hypothesis (rich conditioning becomes a shortcut). |
| `no-empty-cond` | Drop empty-conditioning samples. Tests whether empty conditioning is needed for disentangling conditioning from input. |
| `random-conditioning` | At inference, replace user-specified features with random features of similar magnitude. If the model treats conditioning as SAE-grounded, this should sharply degrade output; if it treats conditioning as an opaque index, no degradation is expected. **A central validity check for SAE-LEWIS.** |
| `single-token-ins-only` | Restrict INS data to single-token spans. Tests the contribution of multi-token training to the editor and the length predictor. |
| `no-compound` | Disable compound corruption; train on identity + single-op buckets only. Tests whether compositional multi-op training is needed for the editor to generalise to real (multi-op) inference-time edits. |
| `uniform-ins-position` | Disable priority-biased INS position selection; use uniform random word positions. Tests whether the priority bias is needed to keep the compound distribution close to natural sentence variation. |
| `fixed-thresholds` | Replace N-dependent gates with constant `ppl_max_ratio` and `sae_shift_threshold` (no `sqrt(N)` scaling). Tests whether N-dependent calibration is required, and whether the resulting distribution shift in compound N actually hurts the editor. |
| `length-predictor` | Replace enumeration with length-head + `±1`. Quantifies the efficiency / quality trade-off. |
| `proj-a-learned` | Train with `--proj-a-mode learned` (random-init Proj_A, the pre-v3.1 configuration). Isolates the contribution of W_dec grounding to feature-identity use — the Δ(true−random) probe. |
| `no-partial-records` | Regenerate with `--emit-partial 0 --emit-null 0` (v2-style cache) and retrain. Isolates the contribution of condition-selective records to conditioning causality — the central v3 claim. |
| `frozen-backbone` | Train tagger and editor with `--lora-r 0` (no backbone adaptation; only Proj_A / heads / delta rows learn — the pre-LoRA configuration). Tests whether backbone adaptation is required for the models to *use* the conditioning: with everything frozen, the only trainable path from the prefix to the edit positions is the linear `Proj_A`, and held-out probes under that configuration scored the editor conditioning-IGNORED. |

### 13.3 LLM2Vec checkpoint evaluation

Run `python eval_llm2vec.py --llm2vec-dir <ckpt> --output-dir <out>` (or `scripts/eval_llm2vec.sh`) to evaluate an MNTP'd checkpoint along five orthogonal axes. The first four are pipeline-specific sanity checks; the fifth mirrors the LLM2Vec paper.

| Axis | What it measures | Failure mode it catches |
| --- | --- | --- |
| **(1) MNTP held-out loss / ppl** | Canonical 15% / 80-10-10 token-level masking + `GemmaForCausalLM.forward(labels=...)` (with its `+1` shift) on held-out Dolma. Loss averaged over non-ignored labels. | MNTP training never converged. Random init ≈ `log(V) ≈ 12`; healthy MNTP lands in the 2-6 range on Dolma. |
| **(2) Causal PPL drift** | Standard next-token PPL of LLM2Vec ckpt vs base LLM, both in **causal** mode, on the same held-out sentences. | The PPL scorer used by `corruption.py` has drifted far from base Gemma's, which would silently shift per-N ratio calibrations. |
| **(3) Bidir-vs-causal divergence** | Same weights forwarded twice — with and without the bidirectional patch. Per-position cosine of final hidden states. Last position should be ≈ 1.0 (no right context to gain); first position should drop materially. | Bidirectional patch silently failed (e.g., SDPA path overrode it). Flagged automatically when mean cosine > 0.999. |
| **(4) Special-token embedding sanity** | Row norm and nearest-neighbour cosine for `[MASK]`, `[INS]`, `[DEL]`. | Pipeline-critical special tokens are under/over-trained. Flagged when norm ratio falls outside `[0.5, 2.0] × median`. |
| **(5) MTEB sentence-embedding eval** | Wraps the bidir model with a chosen `--pooling {mean, last, weighted_mean}` strategy and runs `--mteb-tasks` (default `STSBenchmark`, Spearman correlation). Pooling is the LLM2Vec paper's key ablation axis (§4.3). | General-purpose embedding quality is degraded relative to the paper's expectations. Requires `pip install mteb`; auto-skipped when missing or `--skip-mteb`. |

Outputs: `eval_metrics.json` (machine-readable) + `eval_report.md` (human-readable). MTEB per-task JSONs land under `$OUTPUT_DIR/mteb/`.

### 13.4 Per-N corruption measurement

The compound-N gate's `sqrt(N)`-scaled thresholds (§6.2.6) are calibrated against the empirical Dolma + MLM distribution. To resample this distribution after a corruption / MLM / encoder change, set `MEASURE_COMPOUND_N=1` on `scripts/smoke_pipeline.sh` (stages 7+8 of the smoke). For each N in `MEASURE_N_VALUES`, the smoke runs `corruption.py --force-n $N --calibration-mode` (overriding bucket sampling so every attempt records `(N, ppl_ratio, sae_shift)` regardless of the gate) and then calls `scripts/analyze_compound_n.py` to compute percentiles and simulate gate yield under both the default constants and a grid sweep over `ppl_per_op_factor` / `sae_per_op_min` / `sae_per_op_max`.

Artifacts (`$RUN_DIR/measure_n/`):
- `n{N}/calibration.jsonl` per-attempt records
- `report.tsv` per-N percentile + per-grid yield summary
- `report.md` human-readable version

### 13.5 Tagger / editor held-out evaluation

`eval_tagger_editor.py` runs the per-stage metrics of §13.1 plus a conditioning-causality probe on a **dev corruption cache** — generated from Dolma sentences disjoint from the training cache:

```bash
# 1. Dev cache: same seed as training (identical stream order), skipping
#    past the sentences training consumed (meta.json / "sentences_seen").
SEEN=$(python -c "import json;print(json.load(open('runs/prod/corruption/meta.json'))['sentences_seen'])")
python corruption.py --out-dir runs/prod/corruption_dev \
    --llm2vec-dir <merged ckpt> --target-samples 3000 \
    --seed 42 --skip-sentences "$SEEN"

# 2. Evaluate both models (either ckpt can be omitted)
python eval_tagger_editor.py \
    --corruption-dir runs/prod/corruption_dev \
    --llm2vec-dir <merged ckpt> \
    --tagger-ckpt runs/prod/tagger/tagger-final.pt \
    --editor-ckpt runs/prod/editor/editor-final.pt \
    --output-dir runs/prod/eval_tagger_editor
```

**Level 1 (held-out accuracy).** Tagger: op-head per-class P/R/F1 (KEEP/REPL/DEL), confusion matrix, macro-F1 against the all-KEEP baseline (which the tagger must beat — plain accuracy is dominated by KEEP), insert-head binary P/R/F1, edit-span IoU over the union of both heads' edit positions. Editor: argmax accuracy over the template segment split by position type (KEEP identity, REPL/INS restoration top-1/5) and the marker-emission rate (a special token winning an argmax is always wrong in v2).

**Level 2 (conditioning causality).** Every sample is evaluated under three paired conditions: `true` (diff-based z, as trained, empty-prob 0), `empty` (z = 0), and `random` (same k and same magnitudes as `true` at random feature indices). Verdicts per probe:

| Signal | Reading |
| --- | --- |
| Δ(true − empty) ≈ 0 | conditioning IGNORED — check `cond_scale` (→0 = model muted the prefix), Proj_A, or over-strong sub-sampling (C11) |
| Δ(true − random) ≪ Δ(true − empty) | OPAQUE-FLAG warning — conditioning works as a generic "something changed" cue, not SAE feature identity; the `random-conditioning` ablation (§13.2) would fail |
| both gaps clearly positive | conditioning is used AND SAE-grounded |

**Conditioning-density sweep (`scripts/sweep_eval_hparams.py`).** The eval-time conditioning hyperparameters are tunable: `--k-top` (diff candidate pool), `--k-amp`/`--k-sup` (`'LO-HI'` uniform draw or fixed int) and the INS threshold. Sweep on the SELECTION split (`corruption_seldev`), confirm the chosen setting with a full three-condition run on the reporting dev split. v4 finding: **`k_top=8, k=8` (take all top-8 diff features deterministically, INS threshold 0.9) dominates the training-matched draw (`k~U{1..4}`, threshold 0.5)** — dev-confirmed tagger macro-F1 0.752→0.797, INS F1 0.499→0.694, editor repl_top1 0.566→0.778, ins_top1 0.544→0.651, with the causality gaps *growing* (e.g. editor repl Δ(t−r) 0.31→0.53). Widening `k_top` beyond 8 hurts: the k features are sampled from the pool, so a larger pool dilutes the draw with weak diff features. Report both the training-matched and best-k settings; the mismatch also motivates widening the training-time draw at the next retraining — done in v5 as `k ~ log-uniform{1..32}`, which removed the dilution regime entirely (§13.6).

Diagnostics reported alongside: `cond_scale`, `‖Proj_A.W‖`, and per-token `delta_emb` norms. Outputs `eval_report.md` + `eval_metrics.json`.

### 13.6 v5 results — K-robust training and the selection→generation handoff

v5 = v4 cache + 200k v4d top-up (new edit-geometry families, 15% composition) trained with `K_TOP=32, k ~ log-uniform{1..32}` (§6.2.10, `train_model.sh`). Findings, in causal order:

- **K-robustness worked; the operating point moved to the densest available spec.** The post-training sweep (grid `k_top × k ∈ {4..64}²` on seldev) is monotone in density — the v4 "larger pool dilutes the draw" regime is gone — and peaks at `k_top=64, k=64`, i.e. use everything stored. Dev-confirmed (4000 samples, reporting split): tagger op3 0.6829 / INS-F1 0.5675, editor repl_top1 **0.9386** / ins_top1 **0.7959**, all probes USED + SAE-GROUNDED with large gaps (editor repl true−random 0.52). Caveats: (a) 64 is both the grid edge and the storage cap (`pool_topk=64`), so "64 is optimal" and "all stored information is optimal" are not distinguishable; (b) the dev split now leads with v4d-family shards, so v4↔v5 numbers are not comparable — the v4 checkpoints would need re-evaluation on the new split for a like-for-like table.
- **The cosine sae_align was structurally dead; the ranker now scores commanded-delta satisfaction (§4.4).** On LinguaLens the old component picked identity on 100/100 pairs (93 of which had edit candidates): its ±0.005 range could never offset a real edit's content+fluency penalty (weighted diffs: sae_align +0.002 vs −0.067). The directional replacement restores a usable scale (identity 0.00, edit candidates mean +0.13, max 1.0) and, at k=64: copy 1.00→0.80, iou 0.00→0.106, first exact match, `empty` copy_rate staying 1.00 (structural: empty spec zeroes the component for every candidate).
- **Best-vs-best on the same dev (v4 at 8/8 vs v5 at 64/64): the editor is the whole v5 win; the tagger only redistributed.** Editor repl_top1 0.7375→0.9386, ins_top1 0.5494→0.7959; tagger op3 macro-F1 0.6831→0.6829 (flat), INS F1 0.5385→0.5675. Per-family (`--per-family`, compare on span IoU — macro-F1 is structurally capped near 0.33 for records lacking a class, e.g. identity records and REPL-free families like ADVPLACE/COMPZR): the v5 tagger gained on the new families (CONTRACT 0.286→0.459, COMPZR 0.077→0.500, PPFRONT 0.330→0.455) while losing on established ones (TENSE 0.874→0.835, DETQUANT 0.977→0.788, NUMBER macro 0.759→0.647) — a fixed 30k-step budget spread thinner over a broader distribution, while the 100k-step editor absorbed both (CONTRACT repl 0.44→0.99). → `train_tagger_100k.sh` retrains the tagger alone at 100k steps, all else identical. **Outcome: 30k was severely undertrained.** At 100k (same dev, 64/64): op3 macro-F1 0.6829→**0.8122**, INS F1 0.5675→**0.7112**, span IoU 0.664→0.826, REPL/DEL precision 0.52/0.32→0.65/0.63 (the over-prediction is largely gone), causality gaps GROWING (op3 Δ(t−e) 0.356→0.485, random span IoU 0.106→0.056). Per-family: established families recovered past v4 (TENSE IoU 0.928, NUMBER macro 0.892, DETQUANT IoU 0.977), trainable new families jumped (CONTRACT IoU 0.459→0.944 with INS F1 0.933, COMPZR →0.893, NONFIN →0.793), and only the data-starved reordering families stay low (ADVPLACE IoU 0.375 / INS F1 0, PARTICLE 0.45, DATIVE n=2) — confirming the rarity diagnosis. `tagger_100k` is the v5 reference tagger from here on; use TAGGER_STEPS=100000 in future production runs. The aggregate 0.68 tagger macro-F1 also carries two composition effects: 63% of dev samples are lexical buckets (macro 0.55–0.62, REPL/DEL precision 0.52/0.32 — edit-site over-prediction) and the v4d families are RARE in the generation stream (PARTICLE 0.25%, QUOTINV absent from 4000 dev records ⇒ ~0.05% of the total 1M cache) — their low scores are data starvation more than intrinsic difficulty; the generation-side fix (priority pick for rare families in corruption.py) is deferred to the next top-up cycle together with LOFO.
- **Editor headroom at the operating point is spent.** Ceiling re-measured at 64/64 (`--k-amp/--k-sup` now parameterized): REPL top-1 0.9386 vs top-5 0.9877 / top-50 0.9982 — done; INS top-1 0.7959 vs top-5 0.9210 / top-50 0.9663 — ~12pt of ranking-style headroom, low priority given insertion's inherent multi-answer nature.
- **Selection is no longer the bottleneck; generation is.** Of the 20 chosen edits, 13/20 move the commanded SAE features toward the target (positive sae_shift) but only 2/20 improve sim_target (mean Δ −0.033, median 0.000): local single-site fills satisfy feature deltas without composing the coordinated multi-site rewrite the reference requires — the same ceiling the v4 oracle analysis put on selection (+0.035 max). Copy breakdown of the other 80: 7 no edit candidate, 58 all candidates fluency-gated (>0.5 nats drop — keep the gate; letting these through would lower copy_rate but hurt sim_target given the 2/20 hit rate), 15 lost on score. Remaining levers are generation-side: coordinated multi-site decoding, or LinguaLens-FT with premise protection (held in reserve). Confirmed by the 100k-tagger rerun: a +0.13 in-domain tagger jump moved LinguaLens copy 0.80→0.87 and iou 0.106→0.062 — the sharper tagger is more *precise*, so on OOD inputs it proposes fewer sites (the 30k tagger's over-prediction had been accidentally overlapping gold). In-domain module quality is not the OOD constraint.
- **Gold-template probe: the OOD failure is the fill CONTENT, not localization and not decode coordination** (`scripts/lingualens_gold_template_probe.py` — templates built directly from the s1→s2 token alignment, gold sites + gold slot lengths, tagger and enumeration bypassed; three decode modes over the same template). On 190 LinguaLens pairs at k=64: edit-token top-1 is **0.13** (top-5 0.39) and `true` equals `empty` to the fourth decimal — **the conditioning contributes zero content information OOD**, versus in-domain repl true 0.94 / empty 0.47. Sequential decoding (left-to-right or confidence-first, prediction fed back before the next forward) does NOT help — 0.09/0.11 vs parallel's 0.13, at every n_edit bucket — so the independent-parallel-fill architecture is exonerated; even single-edit pairs (n=99, the minimal-pair core) fill at 0.10 vs 0.94 in-domain. Meanwhile gold localization with these junk fills still reaches sim_target 0.70, far above both the input-copy baseline (0.607) and the full pipeline (0.60): **the SAE conditioning generalizes for WHERE (tagger: USED, SAE-GROUNDED on OOD) but not for WHAT (editor: feature→content association memorized on the corruption distribution)**. Implications: decode-order changes and tagger investment are dead ends for OOD; the lever is content-diverse training pairs (real edit corpora / distilled rewrites in the same cache format), with benchmark-targeted FT (premise-protected) as the fallback.
