# Part I — 理論的背景

## 1. 研究の目的と貢献の位置づけ

- **前提(目的)**: LinguaLensと同様、言語学的featureに対応するSAE
  activationsを見つけること(新たな抽出手法は提案しない)。
- **提案システム**: featureごとに同定された活性(spec=同定プールの
  符号付き平均)を信号として受け取り、編集を**Δhとしてresidual streamに
  描画する学習editor**。テキストは凍結LMの生成として出る = do(residual)
  の介入(ReFT類縁)。
- **中心主張**: activationsの因果妥当性は**編集によって**検証されるべき
  であり、editorによる最小対編集の成立がその因果証拠。評価軸:
  (i) exact(厳密なテキスト接地、net=true−random)、(ii) 統合FIC
  (enhancement/ablationの調和平均、judge判定)、(iii) 特異性
  (random spec統制)、(iv) 同定安定性(split-half)。
- 貢献文(02冒頭と同一): featureレベルSAE specに条件付けられた介入
  生成器+編集ベース因果評価(CausalGymの拡張、SAEBench系への評価器追加)。

## 2. 条件付けと介入 — 分水嶺は**出力インターフェース**

| | 条件付け(トークン出力の編集器) | 介入(steer/clamp/**学習editor**) |
|---|---|---|
| 出力 | **トークン列**(編集器が文を書く) | **Δh(embedding)** — 文は凍結LMが書く |
| 因果式 | P(edit \| Z=z) — zは**入力** | P(Y \| do(Z=z)) — 活性側を**書き換える** |
| 検証器 | 学習された編集器(**内生的** — その符号を読むよう最適化済み) | **凍結LM自身(外生的** — テキストは凍結LMの計算が生む) |
| 証明できること | zが編集を特定する**情報**を持つ(読み手相対) | LMの計算が**その方向に応答する**(因果) |

- 判定基準はアーキテクチャではなく**出力インターフェース**(2026-07-17
  決定): 双方向エンコーダや特徴トークン条件付け(EFの機構)は提案editor
  に流用してよい — トークンを直接出力する構成だけが条件付け。
- トークン出力EF系(S1/S2/S3)は因果証明にならず論文から除外
  (コード・runsは履歴として残す)。
- editorの内容はReFT同様「学習される」が、出力チャネルがΔhである限り
  テキスト生成は凍結LMの計算 = 検証器は外生的。specへの特異性は
  random/empty統制とゼロショットOOD(Dolma学習)、介入サイズはノルム
  予算が担保。

## 3. バックドア基準と「外生的検証器」問題

- 評価でZ(spec)は**こちらが設定**する(true/random/empty割付)=入力への
  正真正銘の do(Z)。**バックドア基準は満たす** — Zへの交絡パスは無い。
- 問題は識別ではなく**推定量の帰属先**: 識別されるのは「**この学習済み
  editorを含むシステム**でZが編集を起こす」であって、「ZがLM/SAE幾何に
  おいてfeatureに対応する」ではない(輸送可能性・構成概念妥当性の問題)。
- 学習された読み手は**単射に近い符号なら何でも**読めるようになる —
  効果の存在が符号(SAE)の性質を弁別しない。
- **SAEのdecodeとの違い**: W_decは再構成のみ学習した**基底変換**で、
  出力は凍結LMの下流計算に食われる(検証はLMが行う)。編集器のdecodeは
  **タスク解読器**(成功するように最適化される)。
- 実例(自前の実測): universal_quantifiers の AUROC=1.000 latent
  (ラベル "proper names and titles")— 読み手ベース評価はこの相関latentを
  「対応」と誤認しうるが、介入(AUROC-r1 steerのexact崩壊)は暴いた。
- **学習editorでも帰属が守られる理由**:
  (i) 出力はΔhのみでテキストは凍結LMが生成(検証器は外生的)、
  (ii) ノルム予算で「介入サイズ」をsteer描画と同桁に拘束(自由な
  soft-prompt符号チャネル化を防ぐ)、(iii) specは**featureレベルで
  オフライン構築・入力独立**(評価ペアの情報を含まない — 07-22の
  プロトコル移行でこの担保は旧事例レベル仕様より強くなった)、
  (iv) random/empty統制とゼロショットOODがspec特異性を担保。

## 4. 「編集が成功したlatent集合」の呼び方と射程

- 選ばれた活性の正式な呼び方: 「**介入によって因果的に検証された
  活性集合**」。feature-spec移行により集合はfeatureレベル(プール平均
  top-64)なので、旧「操作ハンドル(事例内容混入)」問題は大幅に縮小 —
  それでも「k=64がfeatureの表現幅」とは書かない(kは介入の操作幅)。
- featureごとの**最小因果サポート**(どのlatentが本当に担うか)は
  現在計画中の正規ルートで主張する(04§9p): A=t統計ゲート(同定プール
  統計から創発するk)× E'=editor読み出しの必要性テスト(遮蔽/勾配帰属)
  — AとE'の一致集合のみを「サポート」と呼ぶ(循環回避)。

## 5. 検出≠因果編集の乖離(3ドメイン一致)

> **相関的に同定できる ≠ 因果的に編集できる**

- 我々の実測(同一同定プール、L12 ablation exact): FRC-r3 0.016 /
  AUROC-r1 0.070 vs 学習介入 0.128 — 検出的選択の介入は編集を実行できない。
- 先行との並置: Hase et al.(重み編集)、AxBench SAE-A(検出+32%で
  steering−5%)、我々(テキスト編集)— 3ドメイン一致(詳細04§9、05§1a)。

## 6. 情報軸(条件付け評価)の扱い — 不採用の記録(変更なし)

- 対応関係(z入力→編集器→出力)の評価は「情報的忠実性」として成立し得る
  が、SAEの手柄と主張するには**random辞書対照**(同一レシピで辞書だけ
  ランダム差し替えて再学習)が必須。コストと焦点の判断で不採用のまま。

## 7. 用語と免許の一覧(2026-07-23更新)

| 書いてよい | 書いてはいけない |
|---|---|
| 「同定活性への学習介入が編集を実行(因果的検証)」 | トークン出力EF系の成績を論文に載せる(⚫) |
| 「検出的選択(FRC/AUROC)の介入は編集を実行できない」 | 「介入ベースSAE評価は初」(SAEBench先行、規則4) |
| 「mean集約specは選択不安定を解消(0.83 vs 36-43%)」 | 「k=64がfeatureの表現幅」 |
| 「特異性: random specでほぼ無編集」 | 検出的な意味での「対応」とすり替える |
| 「統合FICでpromptingを上回る」(exactでは及ばないと併記) | 「promptingに全面的に勝つ」 |
| enhancement / ablation(規則5') | amp / sup(本文) |
| 「reliabilityを測る」(judge) | 「judge品質/validityを測る」 |
| oracle値を付録で上界診断として引く | oracle値を本文に書く(規則0') |
